# 内部审计记录 — 2026-07-14

保留本记录是为了使项目边界变化可审计。

## 初始仓库状态

- 分支：`main`，跟踪 `origin/main`；初始 HEAD：`15a8ce2`。
- 远程：拉取与推送均为 `git@github.com:LiPume/Zhijia-Guardian.git`。
- 初始工作树干净；编辑前创建了本地保留分支 `legacy-before-openpilot-recalibration`。
- 仓库原先是 CARLA / nuScenes / nuPlan / SafeBench 优先的研究原型，包含生成媒体、论文 PDF、benchmark 表和脚本；它们并不构成同一条物理 ADS route 或真实 CAN/控制链。

## 旧数据流

旧 adapter 将手工场景、CARLA record、nuScenes 观测、nuPlan 扰动和 SafeBench export 转换为 `ScenarioRecord`。顺序图随后执行 scene / perception / planning / control / metric / root-cause / report 节点。输出契约和 failure-sample package 有复用价值，但输入是异构数据集，并非同一消息时间线；部分评估脚本还会把注入标签或场景元数据与生成假设比较。

## Agent 评估

`PerceptionAgent`、`PlanningAgent`、`ControlAgent`、`MetricAgent` 和大量 root-cause 路径本质上是被 Agent 命名包装的确定性规则/指标函数。旧图主要是固定顺序，不具备 Agent 私有工具注册表或 evidence-bounded 路由。`ReportAgent` 和 Pydantic 输出契约的思想可保留，但旧角色命名不可继续作为主线。

## 可复用资产

- Pydantic schema 约束与 observed/oracle 隔离。
- Evidence、diagnosis 与 failure package 的输出思想。
- Synthetic 扰动与 smoke test 模式。
- 既有 package 布局与 pytest 使用方式。

## 冲突、风险与处理

- 旧 README、实验报告、视觉 demo、CARLA/nuScenes/nuPlan 映射与 benchmark 叙事过于宽泛，和新的 openpilot 消息链范围冲突。
- 命名 adapter 暗示不同数据源可组成同一诊断叙事，这是无效的端到端路线主张。
- 旧路径与运行脚本绑定 CARLA、数据集或用户机器；大媒体和论文 PDF 不应留在运行仓库。
- 旧结果表和 demo 既不是新 workflow，也不能构成可复现 openpilot 评估，已从活动树移除。原因见 `docs/legacy_recalibration.md`。

## 核心概念处理

- `ScenarioRecord`：由 `DiagnosticCase` 替代；未来如需迁移，只能作为 legacy adapter，不能进入主流程。
- Evidence：保留为稳定的结构化记录，含来源、topic 与时间引用。
- Report：保留，但只渲染审计后的结构化状态。
- Failure sample package：保留为轻量输出，含 manifest、diagnosis、evidence、trace 与限制，不含原始日志。

## 重校准结果

活动树已围绕新边界重建，旧树只保留在 `legacy-before-openpilot-recalibration`。使用了位于 `/data5/lzx_data/Zhijia-Guardian/reference/openpilot` 的稀疏上游参考实现，以及一个官方 OpenPilotCI qlog smoke 样例。qlog 已成功解析，但其抽样 topic 集不能被描述为完整 CAN/控制诊断。

项目作为受限工程/研究原型值得继续：它能为消息链排查产生可检查、可复现的证据。若重新包装为通用事故根因自动化，或主要以 synthetic 分类准确率衡量，则不值得继续。
