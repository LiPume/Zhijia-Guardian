# Internal Audit Record — 2026-07-14

This record is intentionally kept with the refactor so that the changed project boundary is auditable.

## Initial repository state

- Branch: `main`, tracking `origin/main`; initial HEAD: `15a8ce2`.
- Remote: `git@github.com:LiPume/Zhijia-Guardian.git` for fetch and push.
- The working tree was clean. A local preservation branch, `legacy-before-openpilot-recalibration`, was created before edits.
- The repository contained a CARLA/nuScenes/nuPlan/SafeBench-first research prototype, generated media, paper PDFs, benchmark tables, and scripts. These do not constitute a single physical ADS route or a real CAN/control chain.

## Actual old data flow

Adapters transformed manually generated scenarios, CARLA records, nuScenes observations, nuPlan perturbations, and SafeBench exports into `ScenarioRecord`. A sequential graph then invoked scene/perception/planning/control/metric/root-cause/report nodes. The output contracts and failure-sample packaging were useful, but the inputs were heterogeneous datasets, not one message timeline. Some evaluation scripts compared injected labels or scenario metadata to generated hypotheses.

## Agent assessment

`PerceptionAgent`, `PlanningAgent`, `ControlAgent`, `MetricAgent`, and much of the root-cause path were deterministic rule/metric functions behind agent names. The graph was primarily a fixed sequence, with no agent-owned tool registry or evidence-bounded routing. `ReportAgent` and the Pydantic output contracts are reusable ideas; the old role taxonomy is not.

## Reusable assets

- Pydantic schema discipline and observed-versus-oracle separation.
- Evidence and diagnosis/failure-package output concepts.
- Synthetic perturbation and smoke-test patterns.
- Existing package layout and pytest use.

## Conflicts and risks to remove or downgrade

- README, experiment reports, visual demos, CARLA/nuScenes/nuPlan mappings, and benchmark claims describe a broader perception/planning/control diagnosis story that conflicts with the new openpilot message-chain scope.
- The named adapters implicitly combine unrelated sources into one diagnostic narrative; this is not a valid end-to-end route.
- Paths and runtime scripts are tied to CARLA, datasets, or user machines. Large generated media and academic PDFs should not remain in the operational repository.
- Historic result tables and demos are neither the new workflow nor a reproducible openpilot evaluation and will be removed from the active tree; the rationale is retained in `docs/legacy_recalibration.md`.

## Handling of core concepts

- `ScenarioRecord`: replaced by `DiagnosticCase`; a small legacy adapter may be added later only for migration, never in the primary workflow.
- Evidence: retained as stable, structured evidence records with source/topic/time references.
- Report: retained but rebuilt to render only audited structured state.
- Failure sample package: retained as a lightweight output containing manifest, diagnosis, evidence, trace, and limitations—not raw logs.

## New audit criteria

The new workflow will use a single openpilot-like message timeline, avoid oracle access by agents, state `insufficient_observability` where topics are absent, and report suspected links rather than claiming an unobservable root cause.

## Recalibration outcome

The active tree was rebuilt around the new boundary. The legacy tree remains only on `legacy-before-openpilot-recalibration`. A shallow upstream reference at `/data5/lzx_data/Zhijia-Guardian/reference/openpilot` (with only its small `opendbc` schema submodule initialised) and a single official OpenPilotCI qlog smoke sample were used. The qlog parsed successfully; its decimated topic set is intentionally not presented as a complete CAN/control diagnosis.

The project remains worthwhile as a constrained engineering/research prototype: it creates inspectable, reproducible evidence for message-chain triage. It is not worthwhile if reframed as universal incident root-cause automation or judged primarily by synthetic classification accuracy.
